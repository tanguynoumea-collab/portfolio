<objective>
Wire next-intl v4.12 into the Next 16 scaffold so that:
1. URL `/` redirects to `/fr` (francophone default per D-14) or `/en` based on `NEXT_LOCALE` cookie (D-15) or `accept-language` header (D-14).
2. Locale-prefixed routes `/fr/*` and `/en/*` render with their respective message catalogs.
3. The Next 16 file rename is honored — **`proxy.ts` at repo root, NOT `middleware.ts`** (this is the critical Next 16 breaking change).
4. `messages/fr.json` and `messages/en.json` contain a full key skeleton covering nav, hero, about, projects, skills, contact, footer, palette switcher, errors.404 (ARCH-07).
5. The pipeline is verified by rendering `useTranslations('nav').t('home')` in a stub page and seeing "Accueil" on `/fr` and "Home" on `/en`.
6. `app/[locale]/layout.tsx` includes an explicit `<head>` element (even if empty in Phase 1) as an integration socket for Phase 2's FOUC blocking script (Pitfall #1 mitigation). The `<html>` element carries `suppressHydrationWarning` per Pitfall #1.

Purpose: Every later phase (chrome, sections, projects, 404) renders under `/[locale]` and relies on `useTranslations`. Getting the routing, redirect logic, and message catalog structure right NOW means later phases don't have to revisit i18n architecture. Pitfall #7 (redirect loops, `as-needed` bugs) is mitigated by careful matcher config and the per-CONTEXT.md decision to use `as-needed` with FR fallback (D-17). The `<head>` socket means Phase 2's THEME-05 can drop in the FOUC script without restructuring this layout.

**Deviation from CONTEXT.md D-16 (redirect status code — documented):** CONTEXT.md D-16 specifies a **308** Permanent Redirect for `/` → `/{locale}`. next-intl v4.12's `createMiddleware` ships a **307** Temporary Redirect on the root path with no built-in option to change the status code (verified against next-intl docs at https://next-intl.dev/docs/routing/middleware on 2026-05-25 — no `redirectPermanent`/`permanent`/`status` option exists in the API surface). Switching to 308 would require writing a custom Next 16 proxy wrapper that intercepts next-intl's response and rewrites the status. The impact is negligible for SEO (Google treats 307 and 308 equivalently for canonicalization purposes, per Google Search Central documentation), and for user-facing behavior the two are identical (both redirect, both forward the method). We **accept 307 as the runtime reality for Phase 1**. If SEO analytics later show canonicalization confusion, Phase 7 (deployment polish) or a v2 follow-up can add a custom proxy.ts wrapper that converts 307 → 308. Tracked as v2 deferred work in this plan's deferred notes.

Output: A working bilingual app at `/fr` and `/en` with redirect logic, type-safe message keys, and the canonical message skeleton ready to be filled by Phases 3-6. Plan 01-04 depends on plan 01-03 (transitively 01-01 → 01-02 → 01-03) because Task 3's locale page smoke test asserts the page renders "with Terra cream background and Terra brown text" via `bg-background`/`text-foreground` utilities that only resolve correctly after plan 01-03's shadcn aliasing pass is in place.

NOTE: In next-intl App Router pattern with `[locale]` segment, you have TWO options:
1. Keep `<html>` in root `app/layout.tsx` (works but means root layout sets a static lang); requires `lang` to be set client-side later.
2. Move `<html>` into `app/[locale]/layout.tsx` so `<html lang>` is locale-aware on first paint.

Option 2 is RECOMMENDED by next-intl docs and the architecture research. So:

app/layout.tsx (root passthrough):
```tsx

  return children;
}
```

app/[locale]/layout.tsx (locale-aware layout — has the <html> tag + <head> FOUC socket):
```tsx
import type { ReactNode } from 'react';
import { NextIntlClientProvider } from 'next-intl';
import { setRequestLocale, getMessages } from 'next-intl/server';
import { notFound } from 'next/navigation';
import { hasLocale } from 'next-intl';
import { routing } from '@/i18n/routing';
import '../globals.css';

export function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }));
}

type Params = Promise<{ locale: string }>;

export default async function LocaleLayout({
  children,
  params,
}: {
  children: ReactNode;
  params: Params;
}) {
  const { locale } = await params;
  if (!hasLocale(routing.locales, locale)) notFound();
  setRequestLocale(locale);
  const messages = await getMessages();

  return (
    <html lang={locale} suppressHydrationWarning>
      <head>
        {/*
         * Phase 2 (THEME-05) will inject the palette FOUC blocking script here as either
         * <Script strategy="beforeInteractive"> or inline <script dangerouslySetInnerHTML>.
         * This empty <head> is the integration socket — Pitfall #1 mitigation.
         * The suppressHydrationWarning on <html> above is required because Phase 2 will
         * mutate documentElement.style pre-hydration.
         */}
      </head>
      <body>
        <NextIntlClientProvider locale={locale} messages={messages}>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```
